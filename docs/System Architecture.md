# System Architecture Reference

## Section 1: System Overview

Roster AI is an end-to-end organizational intelligence platform. It ingests unstructured member resumes and project documents, transforms them into structured technical profiles and semantic vectors using IBM watsonx.ai foundation models, and executes global team optimization via the CP-SAT algorithm. All data is persisted locally in a PostgreSQL database with the pgvector extension for high-resolution semantic matching.

## Section 2: Component Breakdown

### Document Ingestion Layer (app/services/ingestion_service.py)

- Purpose: Orchestrates the transition from raw file to database-ready assets.
- Inputs: PDF and Docx files via FastAPI.
- Internal Logic: Performs SHA-256 checks to skip duplicates. Detects native text vs. scanned images to trigger appropriate vision fallback paths.

### IBM watsonx.ai Client (app/intelligence/llm_client.py)

- Purpose: Unified interface to IBM Cloud watsonx.ai models.
- Role: Manages all AI inference and embedding generation.
- Logic: Implements a 2-attempt JSON repair loop to ensure structured data reliability.

### PostgreSQL Data Layer (app/database/models.py)

- Purpose: Local data persistence and vector storage.
- Search: Uses pgvector with HNSW indices for memory-efficient semantic similarity search.
- Dimensions: Stores 1024-dimensional vectors generated by the multilingual-e5-large model.

### Technical Extraction Agent (app/intelligence/agent_service.py)

- Purpose: A LangGraph-based agent that performs "Truth-Anchored" extraction.
- Resumes: Focuses on human name identification and technical skill synthesis.
- Projects: Analyzes slide decks and documents to extract project titles and mission intents.

### Global Matching Optimizer (app/matching/global_optimizer.py)

- Purpose: Solves the constrained assignment problem using Google OR-Tools CP-SAT.
- Distribution: Uses dynamic ranges to ensure even team sizes across all projects.
- Constraints: Respects manual locks (pins) while optimizing the overall organizational fit.

## Section 3: Data Processing Pipeline

1. Extraction: Native text is extracted via Docling; scanned pages are analyzed by Llama 3.2 Vision via watsonx.ai.
2. Synthesis: The LangGraph agent builds a consensus technical profile.
3. Normalization: Extracted skills are mapped against the organizational taxonomy.
4. Embedding: Technical profiles are converted into 1024-dim vectors for the matching engine.
5. Optimization: The CP-SAT solver generates an optimal assignment matrix.

## Section 4: Reliability and Deployment

- Local Deployment: The core stack (PostgreSQL, Redis, FastAPI) is containerized via Docker Compose.
- VRAM Efficiency: Docling and standard processing are pinned to the CPU to reserve resources for the browser and local environment.
- Cache Strategy: Redis caches technical embeddings to minimize IBM Cloud API latency and costs.
